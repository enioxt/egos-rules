+# Skills Playbook — Criar, Vincular e Integrar no EGOS
+
+> Objetivo: transformar o modelo skill-first em rotina operacional para Windsurf, Antigravity e agentes EGOS.
+
+## 1) Quando criar uma nova skill
+
+Crie skill quando houver:
+
+- Tarefa recorrente (>= 2 usos por semana).
+- Prompt longo repetido manualmente.
+- Necessidade de assets/scripts/memória de domínio.
+- Oportunidade de reduzir custo/token em tarefa já conhecida.
+
+Não crie skill quando:
+
+- É uma demanda pontual (one-off).
+- A lógica pertence diretamente ao código do produto.
+- O resultado depende de alta ambiguidade sem padrão reaproveitável.
+
+---
+
+## 2) Estrutura mínima obrigatória
+
+```text
+skills/<skill-id>/
+  skill.md
+  run.py (ou run.ts)
+  examples/
+    input.sample.json
+    output.sample.json
+```
+
+Opcional:
+
+- `memory.json` (se houver estado local explícito)
+- `assets/` (templates, prompts, imagens, exemplos)
+- `references/` (material de domínio, APIs, políticas)
+
+---
+
+## 3) Template de contrato (`skill.md`)
+
+Use este frontmatter como base:
+
+```yaml
+---
+name: web-research
+version: 0.1.0
+description: Pesquisa web com resumo estruturado e fontes citadas
+entrypoint: python run.py
+inputs:
+  - query (string)
+  - locale (string, optional)
+outputs:
+  - summary (string)
+  - sources (list[string])
+side_effects:
+  reads: []
+  writes: []
+  network: ["https"]
+when_to_use:
+  - quando o usuário pedir análise comparativa com fontes
+when_not_to_use:
+  - quando não for permitido uso de rede
+failure_modes:
+  - sem conectividade
+  - fontes com baixa confiabilidade
+---
+```
+
+Boas práticas:
+
+1. `description` deve explicar escopo e limite.
+2. `entrypoint` deve ser executável por terminal local.
+3. `outputs` devem privilegiar JSON para integração com planner.
+4. `side_effects` deve declarar tudo que lê/escreve/chama.
+
+---
+
+## 4) Vincular skill a agentes (matriz agente→skills)
+
+Documentar em tabela no PR da skill:
+
+| Agente | Skill | Motivo | Fallback |
+|---|---|---|---|
+| `ssot_auditor` | `repo-navigator` | localizar arquivos e contratos | busca manual com `rg` |
+| `ui_designer` | `video-ideator` | ideação e variações criativas | resposta textual sem assets |
+
+Regra: se skill já cobre capacidade, evitar duplicar a mesma lógica no agente.
+
+---
+
+## 5) Integração no fluxo operacional
+
+1. Agente/planner consulta `skills/_index.md`.
+2. Seleciona skill por `when_to_use` e tipo de input.
+3. Executa `entrypoint`.
+4. Valida schema/estrutura de saída.
+5. Aplica fallback se falhar e registra motivo.
+
+Checklist por execução:
+
+- [ ] Skill correta foi selecionada
+- [ ] Saída validada
+- [ ] Side-effects esperados
+- [ ] Tempo/custo registrado
+- [ ] Fallback documentado (se houve)
+
+---
+
+## 6) Qualidade mínima (lint + smoke)
+
+Toda skill deve ter:
+
+1. **Lint de contrato**: campos obrigatórios do frontmatter.
+2. **Smoke test**: roda com `input.sample.json` e retorna formato esperado.
+3. **Exemplo de erro**: pelo menos 1 failure mode documentado.
+
+Sugestão de comandos padrão:
+
+```bash
+bun skills:lint
+bun skills:smoke
+```
+
+---
+
+## 7) Padrão para Windsurf e Antigravity
+
+## Windsurf
+
+- Manter skills dentro do workspace.
+- Instrução fixa: “consultar catálogo antes de solução manual”.
+- Comandos curtos (`make skill.<id>` / `bun skill:<id>`).
+
+## Antigravity
+
+- Mesmo catálogo e mesmo contrato.
+- Execução via terminal/tool com `entrypoint`.
+- Parser único para saída JSON.
+
+Regra de ouro: a mesma skill deve funcionar fora do IDE (CLI-first).
+

+
+## 8) Métricas de adoção
+
+Acompanhar por semana:
+
+- Uso de skills por tipo de tarefa.
+- Taxa de sucesso sem fallback.
+- Redução de tempo médio por tarefa.
+- Redução de tokens/custo por tarefa.
+- Skills sem uso (candidatas a arquivamento).
+

+## 9) Definição de pronto (DoD) para PR de skill
+

+- [ ] exemplos de input/output incluídos

+# Plano de Evolução — Skills + Agentes (Windsurf, Antigravity e EGOS)
+
+> Status: Proposta executável (Q1/Q2 2026)  
+> Dono sugerido: Agentic Platform  
+> Última atualização: 2026-02-28
+
+## 1) Contexto e tese de arquitetura
+
+A discussão recente “Agents vs Skills” converge com um aprendizado prático no EGOS:
+
+- **Agente generalista** deve orquestrar (planejar, decidir, validar, explicar).
+- **Skills** devem encapsular expertise reutilizável (instruções + scripts + memória + assets).
+- **Workflows determinísticos** permanecem úteis para automações de alta recorrência/compliance.
+
+Em resumo: manter inteligência central em poucos agentes robustos e mover especialização para um catálogo de skills versionáveis.
+
+---
+
+## 2) Diagnóstico rápido do estado atual no repositório
+
+### Sinais positivos
+
+- Já existe superfície de skills em `.windsurf/skills/`.
+- Já existe superfície de workflows em `.agent/workflows/` e `.windsurf/workflows/`.
+- O projeto já opera com governança SSOT e agentes registrados, o que facilita acoplar um ciclo de qualidade para skills.
+
+### Lacunas atuais
+
+1. **Ausência de contrato unificado para skills** (entrada/saída, side-effects, owner, versão).
+2. **Descoberta fragmentada** (skills espalhadas sem índice único orientado a execução).
+3. **Baixa testabilidade explícita** (faltam smoke tests por skill e validação de schema de output).
+4. **Integração fraca agente→skill** (política “skill-first” ainda não formalizada como padrão único do repo).
+5. **Memória sem padrão** (não há convenção para memória local por skill com guardrails).
+
+---
+
+## 3) Objetivos estratégicos (90 dias)
+
+1. **Skill-first operacional**: todo agente/planner consulta catálogo de skills antes de resolver do zero.
+2. **Portabilidade real**: skills executáveis por CLI em Windsurf, Antigravity e runner local.
+3. **Confiabilidade**: cada skill com contrato + exemplos + smoke check automatizado.
+4. **Governança leve**: versionamento sem atrito, revisão clara e métricas de adoção.
+
+---
+
+## 4) Princípios de desenho
+
+1. **Contrato antes de implementação**: `skill.md` define o que a skill faz e como chamar.
+2. **CLI como interface mínima comum**: toda skill deve rodar via comando determinístico.
+3. **JSON como lingua franca de integração**: input/output estruturado sempre que possível.
+4. **Side-effects declarados**: o agente precisa saber o que a skill lê/escreve/chama.
+5. **Fail-safe**: se a skill falhar, agente explica fallback e registra razão.
+
